# Forecasting Framework: прогнозы, риски и фиксация фактов

Этот файл задает общий метод для всех выпусков курса.

## 1. Не прогнозировать голую метрику
Прогноз должен отвечать на бизнес-вопрос:
- сколько пользователей дойдут до ценности;
- сколько вернутся;
- сколько заплатят;
- сколько денег вернется;
- сколько клиентов потеряем;
- где риск ошибки максимален.

## 2. Считать три сценария
```text
Optimistic Forecast = Base Forecast без понижающих коэффициентов

Base Forecast = историческая метрика по похожей когорте * будущий объем

Conservative Forecast = Base Forecast * K_total
K_total = K_data * K_segment * K_behavior * K_execution * K_seasonality * K_revenue
```

## 3. Универсальная формула риска
```text
Risk Coefficient = 1 - Probability_of_Risk * Impact_of_Risk
```

Пример:
- вероятность риска: 30%;
- impact: если риск случится, прогноз упадет на 40%;
- coefficient = 1 - 0.30 * 0.40 = 0.88.

## 4. Быстрая шкала haircut
| Коэффициент | Когда использовать |
|---:|---|
| 0.95 | данные чистые, когорта похожая, поведение стабильно |
| 0.85 | есть средний риск: новый канал, неполный rollout, сезонность |
| 0.70 | высокий риск: мало данных, меняется сегмент или UX |
| 0.50 | критический риск: прогноз почти не подтвержден фактами |

## 5. Фиксировать факты отдельно от выводов
Факт не должен зависеть от интерпретации аналитика. События, платежи, состояния подписки, версии продукта и сегменты нужно хранить так, чтобы другой аналитик мог пересчитать метрику и получить тот же результат.

## 6. Минимальный набор журналов
- `event_log`: пользовательские события.
- `state_log`: изменения состояний: подписка, аккаунт, заказ, тариф.
- `payment_log`: платежи, возвраты, комиссии, налоги, failed payments.
- `metric_snapshot`: периодический расчет метрики с numerator, denominator, exclusions.
- `forecast_log`: прогноз, коэффициенты риска, автор, дата, версия модели.
- `decision_log`: какое решение приняли по метрике и когда проверили факт.

## 7. Главный принцип
Прогноз без риска - это желание. Метрика без фактов - это мнение. Факт без решения - это склад данных.
